## MicroVM Execution Pool Deployment (v0.4)

- 文档版本: `20260323-v0.4`
- 目标: 将 `exec_backend=microvm` 落到可运行的执行池, 并可选用于 `remote sandbox service`(形成 remote-microVM 池)。

> 说明: v0.4 的 `microvm` 后端采用 "VM-backed container runtime" 的工程落地方式(通过 Docker Engine 的 alternative runtime/shim 机制)。这让你无需先自研 Firecracker 管控面, 也能获得 microVM 级隔离边界。

---

### 1. 选型与前置条件

#### 1.1 运行平台

- **Linux 优先**: microVM runtime/shim 需要安装在运行 Docker daemon 的机器上。
- 需要硬件虚拟化支持(KVM/VT-x/AMD-V)。云上部署需确认是否支持 nested virtualization。

#### 1.2 Runtime 形态

- 推荐使用 Docker Engine 支持的 containerd shim runtime, 通过 `docker run --runtime <runtimeType>` 选择。
- 示例 runtimeType:
  - `io.containerd.kata.v2` (Kata Containers, VM-backed)

参考:
- Docker Alternative container runtimes 文档(包含 `io.containerd.kata.v2` 示例): `https://docs.docker.com/engine/daemon/alternative-runtimes/`

---

### 2. 验证 microVM runtime 是否可用

在安装并启用 runtime/shim 后, 在运行 Docker daemon 的 Linux 主机上执行:

```bash
docker run --rm --runtime io.containerd.kata.v2 hello-world
```

若成功输出 hello-world, 说明 `io.containerd.kata.v2` shim 已可被 Docker 使用。

---

### 3. text2cli: 本机 microVM 执行池配置

在运行 text2cli 的同一台 Linux 主机上配置环境变量(示例):

```bash
export T2_EXEC_MODE=local-untrusted
export T2_EXEC_BACKEND=microvm
export T2_EXEC_MICROVM_RUNTIME=io.containerd.kata.v2

export T2_EXEC_DOCKER_IMAGE=python:3.12-slim
export T2_EXEC_DOCKER_NETWORK=none
```

说明:
- `microvm` 后端复用 docker 的镜像与网络配置, 但通过 `--runtime` 切换到 microVM runtime。
- 在 `local-untrusted` 下, `host` 后端被默认禁止, 所以 microVM/dcoker/remote 是主要执行池。

---

### 4. text2cli: 自动调度时优先 microVM(可选)

如果你希望在 `auto + scheduler` 下更偏向 microVM, 可以提高安全权重或降低延迟/成本权重:

```bash
export T2_EXEC_BACKEND=auto
export T2_EXEC_SCHEDULER=1
export T2_EXEC_POOL_WEIGHTS=security=8,latency=2,cost=1,hit_rate=0
```

以及可选提供命中率/成功率 hints:

```bash
export T2_EXEC_POOL_HIT_RATES=host=99,docker=98,microvm=99,remote=97
```

---

### 5. remote sandbox service: 远端 microVM 执行池(可选)

如果你希望把 microVM 作为"远程执行池"(更适合多租户/隔离域), 可以:

1) 在 Linux 主机上运行 `t2-sandbox` 服务。
2) 让服务端的 docker runtime 指向 microVM runtime(例如设置 `T2_SANDBOX_DOCKER_RUNTIME=io.containerd.kata.v2`)。
3) 客户端设置 `T2_EXEC_BACKEND=remote` + `T2_EXEC_REMOTE_URL=...`。

注意:
- v0.4 的 remote sandbox service 默认以 docker 作为底座; 通过服务端 runtime 配置可升级为 microVM 底座。
- 这样形成的 pool 在调度上仍体现为 `remote`, 但隔离强度由远端 runtime 决定。

---

### 6. 已知限制与建议

- macOS 上通常无法把 `io.containerd.kata.v2` 安装到 Docker Desktop daemon, 因此 **microVM 后端按 Linux-first 设计**。
- microVM 冷启动通常慢于容器, 建议结合任务风险分级与审批策略使用:
  - 高风险/不可信/无人值守: microVM
  - 中风险/强调兼容性与成本: docker
  - 低风险/单机单用户: host(仅 local-trusted)

## 2. 数据模型

### 2.1 数据表

| 表名 | 用途 | 主键 |
|------|------|------|
| `blobs` | 内容寻址文件存储 | `hash` (SHA-256) |
| `commits` | 提交历史链 | `id` |
| `commit_changes` | 每次提交的路径变更 | `(commit_id, path)` |
| `workspaces` | 工作区/分支头 | `name` |
| `staged_changes` | 工作区暂存变更 | `(workspace_name, path)` |
| `events` | 审计事件流 | `id` (自增) |
| `snapshots` | 命名不可变快照 | `name` |
| `policy_hooks` | 策略钩子注册 | `id` (自增) |

### 2.2 读取路径 (快照重建)

```
HEAD commit -> parent -> parent -> ... -> root
         |           |           |
    commit_changes  commit_changes  commit_changes
```

对每个路径, 取最近一次出现的操作 (upsert 取内容, delete 则不显示). 然后叠加暂存区变更. 时间旅行读取绕过暂存区, 直接从指定 commit 回溯.

### 2.3 内容寻址

```
content -> SHA-256(content) -> blob_hash
blob_hash -> blobs 表 -> 实际内容 (仅存一份)
```

## 3. 存储引擎

### 3.1 连接管理

每个高级操作获取独立 SQLite 连接, 操作完成后自动提交或回滚:

```python
@contextmanager
def _connect(self):
    conn = sqlite3.connect(self.db_path)
    conn.execute("PRAGMA foreign_keys = ON")
    conn.execute("PRAGMA journal_mode = WAL")
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:
        conn.close()
```

保证: 无长寿命连接, 异常安全, WAL 并发读取.

### 3.2 提交流程

1. 计算有效变更 (过滤 no-op)
2. 评估策略钩子
3. 发出 `pre_commit` 事件
4. 写入提交记录和变更行
5. 推进 HEAD, 清除暂存
6. 发出 `post_commit` 事件

### 3.3 合并流程

1. 收集源工作区自分叉点以来的变更
2. 收集目标工作区自同一分叉点的变更
3. 检测同路径不同变更 -> 冲突
4. 无冲突则创建合并提交, 更新源工作区 base

### 3.4 缓存策略

```
请求 -> WorkspaceCache -> CacheBackend(Redis/LRU)
                              -> hit -> 返回
                              -> miss -> SQLite -> 写回缓存 -> 返回
```

| 层级 | 缓存键 | 失效策略 | 说明 |
|------|--------|---------|------|
| L1 blob | `t2:blob:{hash}` | 永不主动失效 | 内容寻址不变性 |
| L2 file | `t2:file:{ws}:{path}` | write/delete/commit/rollback | 路径级 |
| L3 state | `t2:state:{ws}:{kind}` | 工作区任何变更 | 状态级 |

后端选择: 有 `T2_REDIS_URL` 用 Redis, 否则本地 LRU (4096 条目, `OrderedDict + Lock`).

核心原则:
- L1 不主动失效 (blob 是 SHA-256 内容寻址)
- 写时失效, 非写时更新 (避免 cache stampede)
- 缓存后端异常静默降级为 miss

## 4. Agent 架构

### 4.1 LangGraph ReAct Agent

```
用户自然语言
    |
    v
LangGraphAgent.handle_message_stream()
    |
    v
create_react_agent(model, tools, prompt)
    |
    v
LLM Call #1 -> AIMessageChunk (token 级流式)
    |           -> tool_call_chunks (工具调用意图)
    v
Tool Execution -> ToolMessage (执行结果)
    |
    v
LLM Call #2 -> ... (多轮循环直到无 tool_calls)
    |
    v
Final AIMessage -> reply (最终回复)
```

### 4.2 流式实现

使用 LangGraph `stream_mode="messages"`, 获取 token 级事件:

- `AIMessageChunk.content` -> 逐 token yield `{"event": "token"}`
- `AIMessageChunk.tool_call_chunks` -> 累积工具调用参数
- 首个 `ToolMessage` 到达时 -> flush 累积的 tool_calls, yield `{"event": "tool_call"}`
- `ToolMessage.content` -> yield `{"event": "tool_result"}`
- 最后一批 tokens (无 tool_calls) -> `done` 时组装为 reply

### 4.3 LLM 多厂商支持

通过 `create_chat_model()` 工厂函数:

| `LLM_PROVIDER` | 实现 | 说明 |
|----------------|------|------|
| `openai` (默认) | `ChatOpenAI` | 兼容 GLM/DeepSeek/Moonshot/本地模型, 通过 `base_url` 切换 |
| `zhipuai` | `ChatZhipuAI` | langchain-community 提供 |

### 4.4 可观测性

Langfuse `CallbackHandler` 注入 LangGraph `config["callbacks"]`:
- 每轮 LLM 调用: input/output, token 用量, 延迟
- 每个 @tool 执行: 输入参数, 返回值
- 端到端 Trace: 从用户消息到最终回复

零侵入, 不在业务代码中埋点.

### 4.5 工具层设计

`create_workspace_tools(db, workspace, executor=, search_client=)` 通过闭包绑定 db + workspace:

```python
def create_workspace_tools(db, workspace, *, executor=None, search_client=None):
    @_lc_tool
    def cat(path: str) -> str:
        """读取文件内容"""
        return db.read_file(workspace, path)["content"]

    # ... 21 个基础工具

    if executor:
        @_lc_tool
        def python_exec(code: str) -> str: ...
        @_lc_tool
        def shell_exec(command: str) -> str: ...

    if search_client:
        @_lc_tool
        def web_search(query: str, count: int = 5) -> str: ...

    return tools
```

工具与 Agent 图完全解耦, 可独立测试.

## 5. FUSE 代码执行

### 5.1 架构

```
Agent: python_exec(code="with open('data.csv') as f: ...")
    |
    v
CodeExecutor.execute("main", "python", code)
    |
    v
FuseManager.ensure_mounted("main")
    -> /tmp/t2-fuse/main/ (FUSE 挂载点)
    |
    v
subprocess(python3 -c code, cwd=/tmp/t2-fuse/main/)
    |
    v
FUSE 拦截 open("data.csv")
    -> WorkspaceFUSE.read()
        -> WorkspaceDB.read_file("main", "data.csv")
            -> SQLite 查询 -> 返回内容
```

### 5.2 FUSE 操作映射

| POSIX 调用 | FUSE 方法 | WorkspaceDB 调用 |
|-----------|-----------|-----------------|
| `open(path, "r")` | `read()` | `read_file(ws, path)` |
| `open(path, "w")` | `write() + flush()` | `write_file(ws, path, content)` |
| `os.listdir(dir)` | `readdir()` | `list_files(ws)` 按前缀过滤 |
| `os.stat(path)` | `getattr()` | `list_files(ws)` (TTL=2s 元数据缓存) |
| `os.remove(path)` | `unlink()` | `delete_file(ws, path)` |
| `os.rename(a, b)` | `rename()` | `read + write + delete` |

### 5.3 安全边界

- 执行模式:
  - `local-trusted`(默认): 宿主机 subprocess, 但做最小硬化.
  - `local-untrusted`: 以不可信代码治理.
    - 默认禁止 `exec_backend=host` 在宿主机执行.
    - 需要切到 `exec_backend=docker`(Linux 优先) 或 microVM/远程 sandbox 后端后才允许执行.
- subprocess 强制 timeout (默认 30s, 最大 120s, 可配置)
- 输出截断(硬上限):
  - stdout/stderr 各自最多保留 50,000 字符, 超出部分丢弃, 避免内存被刷爆
- 环境变量最小暴露:
  - 默认剔除敏感环境变量(LLM/Langfuse/Brave/Redis 等 key)
  - `local-untrusted` 下启用严格 allowlist
- 资源约束(若平台支持):
  - 进程数上限、打开文件句柄上限、单文件写入大小上限
- FUSE 仅暴露单个 workspace (无法跨 workspace 访问)
- macFUSE/libfuse 未安装时, `FuseManager.__init__` 直接 `raise RuntimeError`, python_exec/shell_exec 工具不注册

## 6. T2Script 引擎

### 6.1 编译管线

```
源代码 -> Lexer (tokens.py) -> Token 流
    -> Parser (parser.py) -> AST (ast_nodes.py)
    -> Interpreter (interpreter.py) -> 执行结果
```

### 6.2 模块职责

| 模块 | 行数 | 职责 |
|------|------|------|
| `tokens.py` | 297 | 词法分析: TT 枚举, Token 类, tokenize() |
| `ast_nodes.py` | 155 | AST 节点: dataclass (表达式/管道/语句/Program) |
| `parser.py` | 464 | 递归下降解析器: Token 流 -> Program |
| `interpreter.py` | 539 | 树遍历解释器: ScriptContext + 限制执行 |
| `builtins.py` | 588 | 38 个内置命令注册表 |

### 6.3 运行时隔离

每次执行创建独立的 `ScriptContext`:
- 独立变量作用域
- 独立步数/时间计数器
- 绑定到特定 workspace
- 线程安全 (不共享可变状态)

## 7. 并发模型

### 7.1 SQLite WAL 模式

- 并发读取: 多个 Agent 同时读取不同工作区
- 单写者串行化: 写操作通过 SQLite 内部锁排队
- 每个高级操作是一个原子事务

### 7.2 工作区级隔离

- 每个 Agent 操作自己的工作区
- 暂存区按工作区隔离
- 合并是唯一的跨工作区操作

### 7.3 扩展路径

替换 SQLite 为 FoundationDB/TiKV:
- 元数据层: 分布式事务 KV, 支持多写者
- 内容层: 对象存储 (S3/MinIO)
- 事件层: Kafka/NATS

## 8. 安全

### 8.1 路径安全

- 所有路径规范化为 POSIX 格式
- 禁止 `../` 父目录遍历
- 禁止空路径或 `.` 路径

### 8.2 执行安全

- `exec.run` 仅允许白名单命令 (wc/sort/head/tail 等)
- FUSE 代码执行: subprocess 有 timeout + 输出截断
- FUSE 仅暴露单个 workspace

### 8.3 策略执行

- `policy_hooks` 表声明式策略
- 事务内评估, 失败则整个事务回滚
- 按工作区模式匹配, 细粒度控制

## 9. HTTP 服务设计

### 9.1 架构

`web.py` 使用 Python 标准库 `http.server.BaseHTTPRequestHandler + ThreadingHTTPServer`:

```
Text2CLIApp (应用对象)
    |- WorkspaceDB (存储内核)
    |- WorkspaceAgent (正则命令 Agent)
    |- LLMWorkspaceAgent (LangGraph Agent)
    |- FuseManager + CodeExecutor
    |- BraveSearchClient
    |- WorkspaceCache

Handler (HTTP 请求处理)
    |- _dispatch() -> 路由分发
    |- _handle_chat_stream() -> SSE 流式
    |- _serve_static() -> 静态资源
    |- _write_json() -> JSON 响应
```

### 9.2 选择标准库的原因

- 零外部依赖 (无 Flask/FastAPI)
- 易于容器化
- POC 阶段够用
- 后端替换不影响上层

### 9.3 连接错误处理

`_serve_static()` 和 `_write_json()` 中 `wfile.write()` 包裹 `try/except (BrokenPipeError, ConnectionResetError)`, 客户端断开不崩溃.

## 12. 关键设计决策记录 (ADR)

### ADR-001: CLI 而非 SQL

用 CLI 动词而非 SQL 作为 Agent 接口. LLM 天然具备 shell/CLI 调用能力, 迁移成本最低. CLI 动词集精确控制 Agent 操作边界.

### ADR-002: 内容寻址存储

SHA-256 哈希为键, 自动去重/不可变/可校验.

### ADR-003: 暂存区 + 原子提交

写操作先入暂存区, 保证 Agent 半成品不影响其他读者, 提交时整体评估策略钩子.

### ADR-004: 白名单 exec

`exec.run` 仅允许预定义文本处理命令, 无副作用. 通用代码执行通过 FUSE + subprocess 隔离.

### ADR-005: T2Script 进程内解释器

替代 Docker/VM 的轻量脚本引擎, 每 Agent ~2MB RAM, 语言级隔离, 精确资源控制.

### ADR-006: SQLite 作为 POC 后端

零依赖, 验证 MVCC/事务/冲突检测语义. CLI 层与存储层解耦, 后端可替换.

### ADR-007: 三级缓存

利用内容寻址不变性 (L1 永不失效), 写时失效策略, Redis/LRU 双后端.

### ADR-008: LangGraph ReAct Agent

从手写循环迁移到 LangGraph `create_react_agent`. 工具层 (workspace_tools.py) 与 Agent 图解耦. Langfuse 零侵入追踪.

### ADR-009: FUSE-only 代码执行 (无降级)

代码执行完全依赖 FUSE, 不提供 materialize 临时目录降级. 未安装 macFUSE/libfuse 时, 代码执行工具不注册, Agent 仍可通过其他 21 个工具正常工作.

### ADR-010: LangGraph-only Agent (无降级)

LangGraph 是唯一 Agent 实现, 不保留遗留 ReAct 循环作为 fallback. 未配置 LLM 时系统启动失败而非静默降级.

### ADR-011: Token 级流式 (stream_mode="messages")

使用 LangGraph `stream_mode="messages"` 而非 `"updates"`, 实现 LLM 输出 token 级别实时推送. 避免用户在 LLM 思考期间看到长时间空白.

### ADR-012: Brave Search API

替代 DuckDuckGo (ddgs) 作为联网搜索后端. Brave API 更稳定, 支持重试, 不依赖第三方库的 decode 稳定性.

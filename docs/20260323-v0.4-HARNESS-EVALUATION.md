### 1. 评估目标与非目标

### 1.1 目标

- 给出 text2cli 当前 harness 的“事实基线”与隐含假设.
- 将 `docs/harness_system.md` 中的 6 条路线, 用同一套维度做优缺点对比.
- 明确 text2cli 当前实现更接近哪几条路线, 以及下一步如何演进到更稳的形态(尽量不破坏现有抽象).

### 1.2 非目标

- 不做具体供应商/产品的采购决策.
- 不在本文中直接改动实现; 本文只输出可执行的工程建议与验收口径.

### 2. 当前实现(事实基线)

本节只陈述“已在仓库中存在的实现事实”, 便于后续对比不漂移.

### 2.1 Harness 的模块边界(按责任拆解)

- 虚拟资源层(virtual workspace):
  - `WorkspaceDB` 作为唯一真相源, 对外提供文件/目录语义与提交/回滚/快照等能力.
  - 通过工具层(`create_workspace_tools`)向 Agent 暴露受控的文件与版本化操作(基础工具数=21).
- POSIX 适配层(FUSE):
  - `WorkspaceFUSE` 将 workspace 映射为 POSIX 文件系统视图(读写/遍历/元数据).
  - `FuseManager.ensure_mounted(workspace)` 按需将 workspace 挂载到临时目录(默认 `<tmp>/t2-fuse/<workspace>/`).
- 执行层(subprocess):
  - `CodeExecutor.execute(workspace, language, code)` 通过 `subprocess.run` 执行 Python 或 bash.
  - 执行进程的 `cwd` 被设置为对应 workspace 的 FUSE 挂载点.
  - `python_exec` / `shell_exec` 工具只是对 `CodeExecutor.execute` 的薄封装, 并将输出做展示截断.
- 语言级批处理层(T2Script):
  - `script(code)` 执行仓库内置 DSL, 用“语言级受限能力”覆盖一部分批量任务, 并天然可审计/可复现(但通用性受限).

### 2.2 当前实现提供的控制点/可审计点

- 路径规范化与防穿越:
  - `WorkspaceDB._normalize_path()` 统一为 POSIX 路径并禁止 `..` 父目录穿越, 作为 DB 层的硬约束.
- 版本化与回滚:
  - DB 层支持暂存/提交/回滚/快照, 便于审计与复盘.
- FUSE 内部只暴露单 workspace:
  - 每次挂载绑定一个 workspace, 从“语义层”避免跨 workspace 的文件访问.
- 代码执行的最小安全边界(当前实现):
  - 超时: `DEFAULT_TIMEOUT=30`, `MAX_TIMEOUT=120`.
  - 输出展示截断: `MAX_OUTPUT_CHARS=50_000`(展示层).
- 工具注册的硬开关:
  - 未安装 macFUSE/libfuse 或未能 import `fusepy` 时, `FuseManager` 构造直接 `raise RuntimeError`, 进而 `python_exec/shell_exec` 不会注册.

### 2.3 当前实现的隐含假设与主要缺口(需要显式化)

以下不是“批评”, 而是把隐含前提摊在台面上, 让后续路线选择有可验证的边界.

- 宿主机隔离强度:
  - `CodeExecutor` 在宿主机直接 `subprocess.run` 执行, 当前没有 OS 级沙箱(Seatbelt/Landlock/seccomp)与容器/microVM 边界.
  - 这意味着执行进程理论上可以:
    - 访问 workspace 之外的宿主机文件(绝对路径或相对 `cwd` 的上级路径).
    - 发起网络请求/扫描内网.
    - 持久化修改宿主环境(安装依赖、写入用户目录等).
- 复现性:
  - Python 解释器、系统工具链、依赖版本均来自宿主环境, 同一任务在不同机器/不同时间可能产生差异.
- 资源治理:
  - 当前主要依赖 timeout, 缺少 CPU/内存/磁盘/进程数/文件句柄数等更硬的配额边界.
- 多租户:
  - 当前形态更接近“单机单用户开发体验优先”; 若上升到多租户或不可信代码, 需要引入更强隔离与统一调度.

### 3. 统一评估维度(用于对比 6 条路线)

为避免“各说各话”, 下表是本文对比的统一尺子:

- 隔离强度: 对宿主机/其他租户/网络的保护边界有多硬.
- 可控性与可审计性: 能否拦截、记录、回放关键操作; 状态是否可追溯.
- 兼容性: 能否运行现成的 CLI、编译器、测试套件、浏览器等生态工具.
- 复现性: 环境是否可声明化与可重放(镜像/快照/依赖锁定).
- 性能与交互性: 启动成本、执行延迟、I/O 开销, 以及本地开发的体验.
- 运维复杂度与成本: 部署、扩容、治理与长期维护成本.
- 多租户与策略治理: 是否天然适配配额、路由、调度与安全策略(控制面/调度面).

### 4. 六条路线优缺点(统一口径对比)

本节对照 `docs/harness_system.md` 的 6 类方案, 结合上面的维度归纳优缺点.

### 4.1 路线 1: 纯虚拟文件系统/虚拟资源层 harness

### 典型形态

- Agent 只获得受控 API: 文件/对象 CRUD, 查询, 版本化, 事件流.
- 不提供任意 shell 或通用代码执行; 或只提供“受限 DSL/受限内置命令”.

### 优点

- 隔离强度: 高(默认没有任意代码执行, blast radius 小).
- 可控性与可审计性: 高(所有操作都走 API, 容易记录与回放).
- 多租户与策略治理: 强(易做 ACL、限流、配额、审计、数据驻留).
- 复现性: 强(状态完全外部化, 回放不依赖宿主环境).

### 缺点

- 兼容性: 弱(难以运行现成工具链; 需要大量“语义补齐”).
- 开发效率: 对复杂任务可能变慢(需要为常用 CLI/编译/依赖管理提供替代物).

### 对 text2cli 的适配结论

- text2cli 当前已经把“工作区状态”做成了纯虚拟资源层(WorkspaceDB + 基础 21 工具), 并额外提供 T2Script 作为“受限批处理”.
- 若把目标锁定在“文本处理/轻量工程操作”, 路线 1 的投入产出比极高.
- 但若目标包含“跑真实项目测试/构建/依赖安装/浏览器自动化”, 路线 1 必须与其他执行后端组合(自然导向路线 6).

### 4.2 路线 2: OS 级受限执行 harness

### 典型形态

- 在本机或近本机直接执行命令, 通过 OS 原生机制做权限与资源约束(沙箱 profile、系统调用过滤、文件访问白名单、网络策略等).

### 优点

- 性能与交互性: 好(启动快、延迟低, 本地体验通常最好).
- 兼容性: 强(能直接跑本机工具链与依赖管理).
- 运维复杂度与成本: 低到中(偏“本地集成工程”而非“基础设施工程”).

### 缺点

- 隔离强度: 中(强依赖 OS 机制与配置质量; 跨平台一致性难).
- 多租户: 弱到中(更适合单机/单用户; 多租户需要更复杂的隔离与审计).
- 复现性: 中(仍受宿主环境影响; 需要补齐环境声明与依赖锁定).

### 对 text2cli 的适配结论

- text2cli 当前的 `subprocess + cwd=FUSE` 更像“路线 2 的一个子形态”, 但缺少 OS 级沙箱与更硬的资源控制.
- 如果要把 text2cli 定位为“单机个人 Copilot/本地 GUI agent”, 补齐路线 2 的 OS 沙箱是最轻量的强化路径.

### 4.3 路线 3: 容器型 harness

### 典型形态

- 为每次任务或每个会话启动容器(或复用长生命周期容器), 在容器内执行命令/安装依赖/跑测试.
- 通过镜像固定工具链与依赖, 用 cgroups/namespace 实现资源与权限隔离.

### 优点

- 兼容性: 强(几乎等同真实 Linux 环境).
- 复现性: 强(镜像+锁定依赖).
- 隔离强度: 中到高(通常显著强于直接宿主机执行; 可选网络隔离/只读根文件系统等).
- 多租户: 中到强(结合 Kubernetes 很自然进入调度面/配额面).

### 缺点

- 性能与交互性: 中(容器启动与文件系统映射带来额外开销; macOS 上容器本质走 VM, I/O 更敏感).
- 安全: 边界强度高度依赖容器配置(卷挂载、capabilities、网络、daemon 暴露等).
- FUSE/虚拟 FS 集成: 需要设计“workspace 如何进入容器”(bind mount、导出/同步、或将 FUSE 挂载点映射进容器).

### 对 text2cli 的适配结论

- text2cli 已存在 Docker 相关文件, 但当前 `CodeExecutor` 仍是宿主机 subprocess.
- 如果你要把执行面从“宿主机执行”迁移到“容器执行”, 推荐的最小破坏方案是:
  - 保留 WorkspaceDB 与 FUSE 语义(状态面不变).
  - 将 FUSE 挂载点作为一个目录 bind-mount 进容器, 在容器内以该目录作为工作目录执行.
  - 容器默认禁网、最小权限用户、只开放必要系统调用与资源配额.

### 4.4 路线 4: microVM/轻量虚拟机型 harness

### 典型形态

- 为执行任务提供更硬的虚拟化边界, 目标是在安全边界上接近 VM, 性能上接近容器.

### 优点

- 隔离强度: 高(虚拟化边界通常比容器更硬).
- 多租户: 强(天然适配“不可信代码”与更严格的合规要求).
- 执行生态: 强(与容器类似, 能跑几乎全部工具链).

### 缺点

- 运维复杂度与成本: 高于容器(需要更复杂的基础设施与镜像/快照管理).
- 性能与交互性: 中(启动成本与资源开销通常高于容器, 但优于传统 VM).
- 本地开发体验: 需要额外工程(尤其是 macOS 开发机上).

### 对 text2cli 的适配结论

- 若 text2cli 的目标升级为“无人值守跑不可信代码/长链路自动化”, microVM 是更稳的隔离面底座.
- 建议把 microVM 作为路线 6(混合式)中的一个可路由 backend, 而不是把所有任务一刀切迁移到 microVM.

### 4.5 路线 5: 远程托管 sandbox/sandbox-as-a-service

### 典型形态

- harness 不在本机, 而是通过 API/SDK 动态申请云端隔离环境, 上传代码/数据后执行, 再拉回结果与日志.

### 优点

- 弹性与集中治理: 强(更容易做多租户、统一审计、统一镜像与配额).
- 运维友好: 执行底座可统一升级与治理(对上层控制面透明).
- 资源可扩展: 易扩到 GPU/大内存/多机.

### 缺点

- 数据驻留与合规: 需要单独评估(尤其涉及源码/客户数据).
- 成本与延迟: 远程调用与数据搬运引入额外延迟与费用.
- 离线/本地体验: 弱(依赖网络与平台 SLA).

### 对 text2cli 的适配结论

- text2cli 的 content-addressed storage 与快照/commit 语义, 天生更容易做“增量上传/去重”, 这对远程 sandbox 很友好.
- 但需要新增一个“执行面协议”: 如何把 workspace 的可见视图安全地投递到远端, 以及如何把结果与变更回写到 WorkspaceDB.

### 4.6 路线 6: 混合式 harness

### 典型形态

- 上层提供统一 harness API(文件/对象 + 执行), 底层按能力/风险/路径路由到不同 backend:
  - 低风险/高可控: 虚拟资源层.
  - 高兼容/执行需求: 容器或 microVM.
  - 高弹性/多租户: 远程 sandbox.

### 优点

- 综合最优: 在“可控性/审计/安全/兼容性”之间做可配置的 Pareto 平衡.
- 演进友好: 可以先用本地后端, 再逐步引入容器/microVM/远程, 而不破坏上层 API.
- 更贴合控制面/执行面/隔离面分离的工程范式.

### 缺点

- 复杂度: 高于单一路线(需要路由策略、状态一致性、调试与观测).
- 一致性风险: 同一任务跨 backend 时, 对 workspace 的视图一致性与并发控制需要显式设计.

### 对 text2cli 的适配结论

- text2cli 当前已经具备“混合式”的关键前提:
  - 状态面外部化(WorkspaceDB).
  - 上层工具层/Agent 图与执行面解耦.
  - FUSE 作为“把虚拟状态映射为 POSIX”的适配器.
- 但当前执行面仍是宿主机 subprocess, 隔离面偏弱. 若目标面向不可信代码或多租户, 推荐把 subprocess 后端替换为容器或 microVM, 并保留统一 API.

### 6. text2cli 当前实现的定位(基于上述路线)

### 6.1 归类

- 更接近路线 6(混合式)的早期形态: 路线 1(虚拟资源层) + 路线 2 的“宿主机执行子形态”.
- 关键特征是: 用 WorkspaceDB 做唯一真相源, 用 FUSE 把虚拟状态投影为 POSIX, 再用 subprocess 执行获得高兼容性.

### 6.2 当前形态的核心优点(对齐你的项目方向)

- 状态与执行解耦: workspace 操作可完全走 DB 工具层, 执行能力可按需启用/禁用.
- 审计与回滚友好: commit/rollback/snapshot 让 “可复盘、可恢复” 成为默认路径.
- 与 Agent 控制面契合: 工具层是明确边界, 便于把“策略选择/任务图”上移到控制面.
- 本地体验好: FUSE + subprocess 让大量现成脚本/工具链可直接运行, 交互延迟低.

### 6.3 当前形态的关键风险(需要按目标场景决定是否接受)

- 若执行输入包含不可信代码或需要多租户:
  - 宿主机 subprocess 不是足够硬的隔离面.
  - timeout/输出截断只能缓解“卡死/刷屏”, 不能构成安全边界.
- 若需要强复现:
  - 宿主机环境漂移会导致结果不稳定; 需要镜像化或依赖锁定.

### 7. 推荐演进路线(尽量不破坏现有抽象)

目标是: 保持“上层统一 harness API + 状态面唯一真相源”的优点, 把隔离面补齐到可扩展的形态.

### 7.1 短期(1-2 周量级): 把隔离假设写清楚 + 最小硬化

- 明确运行模式:
  - `local-trusted`: 单机单用户, 允许宿主机执行(当前默认).
    - 落地: `T2_EXEC_MODE=local-trusted`
  - `local-untrusted`: 本机也要按不可信代码治理(需要更硬隔离).
    - 落地: `T2_EXEC_MODE=local-untrusted`
      - 默认禁止 `T2_EXEC_BACKEND=host` 的宿主机执行
      - 需切到 `T2_EXEC_BACKEND=docker`(Linux 优先) 或 microVM/远程 sandbox 后端后才允许执行
- 最小硬化清单(不改变架构):
  - `CodeExecutor` 的环境变量白名单/黑名单策略(避免泄漏宿主机密钥).
    - 建议默认 deny: `LLM_API_KEY/GLM_API_KEY/ZHIPUAI_API_KEY/LANGFUSE_SECRET_KEY/BRAVE_API_KEY/T2_REDIS_URL` 等
  - 默认禁网或可配置网络策略(即便先从“提醒+显式开关”开始).
    - 关键点: 宿主机 subprocess 很难提供“可信禁网”, 若要强约束建议直接路由到容器/microVM/远程 sandbox 后端
  - 资源约束从“只有 timeout”扩展到“进程数/文件大小/输出大小/工作目录写入范围”等.
    - 建议最小可落地项:
      - 输出大小硬上限(避免刷爆内存/前端): `T2_EXEC_MAX_OUTPUT_CHARS`
      - 进程数上限: `T2_EXEC_MAX_PROCS`
      - 打开文件句柄上限: `T2_EXEC_MAX_NOFILE`
      - 单文件写入上限: `T2_EXEC_MAX_FSIZE_BYTES`

### 7.2 中期(可并行推进): 引入容器后端作为可选执行面

- 以路线 6 的方式接入:
  - `exec_backend=host`(现状) 与 `exec_backend=docker`(新增) 共存.
  - 控制面按任务风险与需求选择 backend.
- 对 FUSE 的最小复用:
  - 继续由宿主机挂载 FUSE.
  - 容器执行时将 FUSE 挂载点 bind-mount 进容器, 容器内 `cwd` 指向该目录.

### 7.3 长期: microVM/远程 sandbox 作为高风险任务的执行池

- 当进入多租户或高权限工具调用阶段:
  - 将 microVM 或远程托管 sandbox 作为“隔离面执行池”, 控制面做过滤+打分+绑定式调度.
  - 仍保持 WorkspaceDB 作为状态面唯一真相源, 并通过内容寻址/快照做高效同步.

落地说明(当前版本):

- 已落地: 远程 sandbox 的接口与执行池/调度骨架(见 `docs/20260323-v0.4-EXECUTION-POOLS.md`).
- 已落地: microVM 执行池(以 Docker alternative runtime/shim 方式落地, 例如 `io.containerd.kata.v2`, 见 `docs/20260323-v0.4-MICROVM-DEPLOYMENT.md`).
- 待落地: Firecracker-native microVM 执行池(不依赖 Docker runtime), 以及基于内容寻址的增量同步.

### 8. 验收口径(如何判断路线升级成功)

- 隔离面:
  - 能证明执行进程无法访问 workspace 之外的宿主机路径(或在策略中被显式拒绝并可审计).
  - 能证明默认网络策略符合预期(默认禁网或白名单).
- 可控与审计:
  - 每次执行都可复盘: 输入、环境、stdout/stderr、退出码、超时与资源消耗可追溯.
- 复现性:
  - 对同一 workspace snapshot, 在不同机器上可重复得到一致结果(在选定的复现范围内).

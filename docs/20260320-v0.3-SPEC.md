## 1. 项目定位

text2cli 是一个 **Agent-native 事务工作区数据库**, 用于替代传统的"每任务一台完整云电脑/完整沙箱"方案. 其核心论题是:

> 数据库伪装成 Agent 所需的最小工作区语义, 而非通用 POSIX 替身.

系统暴露 CLI 命令集(而非 SQL 或 POSIX 系统调用), 让 AI Agent 以为自己在操作文件系统, 实际上所有操作都运行在事务对象图上, 具备 MVCC 快照隔离、原子提交、冲突检测、策略执行和完整审计能力.

### 1.1 与现有方案的对比

| 维度 | 完整 VM 沙箱 (Manus/Codex) | 事务工作区 (text2cli) |
|------|---------------------------|----------------------|
| 隔离方式 | OS 级进程隔离 | 事务级工作区隔离 |
| 启动成本 | VM/容器引导 + 依赖安装 | 零(创建工作区行记录) |
| 代码执行 | 通用 Shell/Python (需隔离) | FUSE 挂载 + subprocess / T2Script (~2MB/agent) |
| 存储成本 | 每任务完整文件系统镜像 | 内容寻址去重, 跨 Agent 共享 |
| 并发能力 | ~10 Agent/机器 | ~1000+ Agent/机器 (工作区级隔离) |
| 审计能力 | 外部日志 | 内建提交日志 + 事件流 |
| 回滚能力 | 快照/恢复 VM | ws.rollback / ws.reset |
| CLI 兼容性 | 完整 shell | 收窄动词集(足以覆盖大部分 Agent 任务) |

### 1.2 适用场景

- 文档处理: Markdown/文本创建、编辑、搜索、组织
- 代码生成: Agent 生成代码后通过工作区提交审核
- 多 Agent 协作: 多个 Agent 在独立分支上并行工作, 最终合并
- 批量数据处理: 结构化文件的批量变换和验证
- 审计场景: 所有操作可追溯、可回放、可回滚

## 2. 核心概念

### 2.1 工作区 (Workspace)

工作区是 Agent 的隔离执行域, 类比 Git 的分支(branch). 每个工作区有:

- **name**: 唯一标识符, 如 `main`、`agent-a`、`task-42`
- **head_commit_id**: 当前最新提交的引用
- **base_commit_id**: 分叉点, 用于合并时的冲突检测
- **tracking_workspace**: 上游工作区(若通过 fork 创建)

关键语义:
- 每个 Agent 应操作自己的工作区, 避免直接修改 `main`
- 工作区创建是即时的(仅插入一条引用记录, 不复制数据)
- 多个工作区可同时对同一底层数据进行独立操作

### 2.2 暂存区 (Staged Changes)

所有写操作首先进入暂存区, 不直接影响已提交的快照. 这保证了:

- 写入可撤回(ws.rollback)
- 提交是原子的(全部或无)
- 其他读者不受未提交写入影响

### 2.3 提交 (Commit)

提交是一次原子状态变更, 包含:

- **commit_id**: 全局唯一标识
- **parent_commit_id**: 父提交引用, 形成链式历史
- **kind**: `root` | `commit` | `merge`
- **message**: 提交说明
- **metadata_json**: 扩展元数据
- **commit_changes**: 此次提交修改的路径列表及操作(upsert/delete)

### 2.4 内容寻址存储 (Content-Addressed Blobs)

文件内容以 SHA-256 哈希为键存储. 相同内容只存一份, 跨工作区、跨提交自动去重.

### 2.5 快照 (Snapshot)

命名快照是对特定提交的不可变引用, 用于:

- 标记里程碑(如 `release-v1`)
- 时间旅行读取的锚点
- 审计基线

### 2.6 事件流 (Events)

所有关键操作自动生成事件记录:

| 事件类型 | 触发时机 |
|---------|---------|
| `workspace_initialized` | 数据库初始化 |
| `workspace_created` | 创建新工作区 |
| `pre_commit` | 提交前(策略钩子评估后) |
| `post_commit` | 提交成功后 |
| `pre_merge` | 合并前 |
| `post_merge` | 合并成功后 |
| `merge_conflict` | 检测到合并冲突 |
| `snapshot_created` | 创建命名快照 |
| `staged_rollback` | 回滚暂存变更 |
| `workspace_reset` | 重置工作区 HEAD |

## 4. CLI 命令规格

所有命令返回 JSON, 退出码规范:
- `0`: 成功
- `1`: 校验错误 / 未找到
- `2`: 合并冲突
- `3`: 策略拒绝

### 4.1 工作区管理

#### `t2 init`

初始化数据库, 创建 `main` 工作区和根提交.

```
输入: 无(仅 --db 路径参数)
输出: { status, db_path, main_head_commit_id }
```

#### `t2 ws.create <name> [--from <workspace>]`

从现有工作区创建分支. 默认从 `main` 分支.

```
输入: name(必填), from_workspace(可选, 默认 "main")
输出: { status, workspace, from_workspace, base_commit_id }
约束: name 不能为空, 不能与 from_workspace 相同, 不能重名
```

#### `t2 ws.list`

列出所有工作区.

```
输出: { status, workspaces: [{ name, head_commit_id, base_commit_id, tracking_workspace, updated_at }] }
```

#### `t2 ws.diff [--workspace <name>]`

显示暂存区与 HEAD 的差异.

```
输出: { status, workspace, changes: [{ path, op, before_blob_hash, after_blob_hash, diff }] }
```

#### `t2 ws.commit [--workspace <name>] -m <message>`

原子提交暂存变更. 流程:
1. 计算有效变更(过滤 no-op)
2. 评估策略钩子
3. 发出 pre_commit 事件
4. 写入提交记录和变更
5. 推进 HEAD, 清除暂存
6. 发出 post_commit 事件

```
输出: { status, workspace, commit_id, parent_commit_id, paths }
错误: 空消息 → ValidationError; 策略拒绝 → PolicyRejection
```

#### `t2 ws.merge --source <ws> --target <ws> -m <message>`

将源工作区的变更合并到目标工作区. 使用乐观冲突检测:
1. 收集源工作区自分叉点以来的变更
2. 收集目标工作区自同一分叉点的变更
3. 检测同路径不同变更 → 冲突
4. 无冲突则创建合并提交

```
输出: { status, source, target, commit_id, paths }
错误: 冲突 → ConflictError(附冲突列表)
```

#### `t2 ws.log [--workspace <name>] [--limit N]`

显示提交历史(沿 parent 链向后遍历).

#### `t2 ws.rollback [--workspace <name>]`

丢弃工作区所有暂存变更.

```
输出: { status, workspace, discarded_count, head_commit_id }
```

#### `t2 ws.reset [--workspace <name>] --to <commit_id>`

重置工作区 HEAD 到指定祖先提交, 同时丢弃暂存.

```
约束: target_commit_id 必须是当前 HEAD 的祖先
输出: { status, workspace, previous_head, new_head }
```

#### `t2 ws.snapshot <name> [--workspace <name>]`

创建命名不可变快照.

```
约束: 快照名不能重复
输出: { status, snapshot, workspace, commit_id }
```

#### `t2 ws.snapshots [--workspace <name>]`

列出快照.

### 4.2 文件操作

#### `t2 fs.read [--workspace <name>] [--at <commit_id>] <path>`

读取文件. 解析优先级:
1. 暂存区(若无 --at)
2. HEAD 提交链
3. 若指定 --at, 直接从该提交解析(时间旅行)

```
输出: { status, workspace, path, content, blob_hash, source, head_commit_id|at_commit_id }
source: "staged" | "committed" | "historical"
```

#### `t2 fs.write [--workspace <name>] <path> --text <content> | --from-file <file>`

写入/创建文件到暂存区.

```
输出: { status, workspace, path, op, blob_hash, size_bytes }
```

#### `t2 fs.patch [--workspace <name>] <path> --append <text> | --find <text> --replace <text>`

原地修改文件(读取当前内容 → 变换 → 写回暂存).

#### `t2 fs.delete [--workspace <name>] <path>`

删除文件(写入暂存区的删除标记).

#### `t2 fs.list [--workspace <name>]`

列出工作区所有文件(已提交 + 暂存叠加).

```
输出: { status, workspace, head_commit_id, files: [{ path, blob_hash, size_bytes, staged }] }
```

#### `t2 fs.find [--workspace <name>] <glob-pattern>`

基于 glob 模式搜索文件路径.

```
示例: t2 fs.find "*.py"
输出: { status, workspace, pattern, matches: [{ path, blob_hash, size_bytes, staged }] }
```

#### `t2 fs.grep [--workspace <name>] [--max-matches N] <regex-pattern>`

基于正则表达式搜索文件内容. 返回匹配行及行号.

```
示例: t2 fs.grep "import os"
输出: { status, workspace, pattern, results: [{ path, matches: [{ line, text }] }], total_matches, truncated }
```

### 4.3 受控执行

#### `t2 exec.run [--workspace <name>] <command> [--path <path>] [args...]`

在工作区上下文中执行白名单文本处理命令. 命令不直接操作磁盘, 而是读取工作区快照中的文件内容进行处理.

白名单命令:

| 命令 | 说明 | 需要 path |
|------|------|----------|
| `wc` | 统计行/词/字符 | 是 |
| `sort` | 排序行 | 是 |
| `head` | 显示前 N 行(默认 10) | 是 |
| `tail` | 显示后 N 行(默认 10) | 是 |
| `uniq` | 去除相邻重复行 | 是 |
| `tr` | 字符替换(需两个参数) | 是 |
| `toc` | 生成 Markdown 目录 | 是 |
| `wordfreq` | 词频分析(Top 30) | 是 |
| `linecount` | 统计所有文件行数 | 否 |

```
输出: { status, command, path?, output }
约束: 非白名单命令 → ValidationError
```

### 4.4 T2Script 脚本引擎

#### 设计动机

`exec.run` 仅支持单条命令执行, Agent 需要多次 LLM 往返才能完成批量操作. T2Script 是一个 **进程内 shell-like 脚本引擎**, 允许 Agent 编写完整脚本一次性执行, 无需 Docker/VM 开销.

| 维度 | Docker/VM | T2Script |
|------|-----------|----------|
| 每 agent 资源 | ~512MB RAM, 0.5 CPU | ~2MB RAM, 共享 CPU |
| 启动延迟 | 125ms-5s | <1ms (函数调用) |
| 隔离 | OS 级 | 语言级 (无 syscall, 无真实 FS) |
| 可回滚 | 需快照 VM | 天然事务 (workspace) |

#### `t2 exec.script --code <T2Script> | --file <workspace-path>`

执行一段 T2Script 脚本. `--code` 传入内联代码, `--file` 从工作区文件加载脚本.

```
输出: { status, output, variables, steps, elapsed }
API: POST /api/exec-script { workspace, code }
```

#### T2Script 语法

```bash
# 变量与类型: string, number, bool, list, map, null
let name = "project"
let items = ["a", "b", "c"]
let config = {host: "localhost", port: 8080}

# 管道 -- 核心组合机制
fs.read "data.csv" | sort | head 5 | fs.write "top.csv"

# 字符串插值
echo "hello $name"

# 控制流
if $count > 10 { echo "big" } elif $count > 5 { echo "mid" } else { echo "small" }

for file in fs.find "*.md" {
    let lines = fs.read $file | wc -l
    echo "$file: $lines lines"
}

# 函数
fn summarize(path) {
    let content = fs.read $path
    let words = $content | wc -w
    return {path: $path, words: $words}
}

# 异常处理
try { fs.read "missing.txt" } catch err { echo "Not found: $err" }

# 模块导入 (从工作区文件)
import "lib/helpers.t2"

# HTTP 外部交互
let data = http.get "https://api.example.com/data"
```

#### 内置命令 (38 个)

| 类别 | 命令 |
|------|------|
| 文本处理 | `echo`, `grep [-v] [-i] [-c]`, `sort [-r] [-n]`, `uniq [-c]`, `head N`, `tail N`, `wc [-l] [-w] [-c]`, `tr`, `replace`, `split`, `join`, `cut [-d] [-f]`, `upper`, `lower`, `trim`, `rev`, `toc`, `wordfreq` |

| WS 操作 | `ws.commit`, `ws.diff`, `ws.rollback`, `ws.snapshot`, `ws.log` |
| 网络 | `web.search`, `http.get`, `http.post` |
| 工具函数 | `len()`, `type()`, `str()`, `num()`, `keys()`, `values()`, `range()`, `append()`, `set()`, `int()`, `float()`, `split()`, `join()` |

## 6. Agent 接口规格

### 6.1 工具 Schema

系统导出 15 个工具 schema, 兼容 OpenAI function-calling 格式:

1. `fs.read` -- 读取文件(支持时间旅行)
2. `fs.write` -- 写入/创建文件
3. `fs.patch` -- 原地修改文件
4. `fs.delete` -- 删除文件
5. `fs.list` -- 列出文件
6. `fs.tree` -- 目录树视图
7. `fs.find` -- 路径搜索
8. `fs.grep` -- 内容搜索
9. `ws.commit` -- 提交变更
10. `ws.diff` -- 查看差异
11. `ws.rollback` -- 回滚暂存
12. `ws.snapshot` -- 创建快照
13. `ws.log` -- 查看历史
14. `exec.run` -- 受控执行(单条命令)
15. `exec.script` -- T2Script 脚本执行(批量操作)

### 6.2 Agent 响应格式

```json
{
  "status": "ok",
  "reply": "人类可读的回复文本",
  "actions": [
    {
      "tool": "fs.write",
      "args": { "path": "...", "content": "..." },
      "summary": { "path": "...", "op": "upsert" }
    }
  ]
}
```

每次 Agent 响应必须包含:
- `reply`: 给用户的自然语言回复
- `actions`: 实际执行的工具调用列表(可为空)

这保证了前端可以追踪 Agent 的实际操作.

### 6.3 LangGraph ReAct Agent 架构

Agent Chat 是**自然语言接口**: 用户用自然语言描述任务, LangGraph ReAct Agent 自主规划并执行 Linux 风格的命令完成任务.

```
用户: "把 a.txt 和 b.txt 的内容合并到 c.txt"
  ↓
LangGraph create_react_agent:
  LLM Call #1 → tool_calls: [cat(path="a.txt")]
    → Observation: "content of a"
  LLM Call #2 → tool_calls: [cat(path="b.txt")]
    → Observation: "content of b"
  LLM Call #3 → tool_calls: [write_file(path="c.txt", content=...)]
    → Observation: "Written to c.txt"
  LLM Call #4 → tool_calls: [commit(message="合并 a 和 b 到 c")]
    → Observation: "[abc1234567] 合并 a 和 b 到 c (1 files)"
  LLM Call #5 → content: "已将 a.txt 和 b.txt 的内容合并写入 c.txt"
  ↓
Langfuse Trace (自动):
  每个 LLM Call 和 Tool 执行均被追踪, 包含 token 用量、延迟、成本
```

核心原则:
- **用户只说自然语言**, 不需要知道任何命令
- **LangGraph 驱动**: 使用 `create_react_agent` 构建 ReAct 图, 自动管理 LLM ↔ Tool 循环
- **工具层分离**: 21 个 `@tool` 函数定义在 `workspace_tools.py`, 与 Agent 图解耦
- **Langfuse 可观测性**: 通过 LangChain CallbackHandler 零侵入追踪所有 LLM 调用和工具执行
- **LLM 多厂商支持**: ChatOpenAI (GLM/DeepSeek/Moonshot/本地) 或 ChatZhipuAI, 通过环境变量切换

### 6.4 Agent 工具集 (ReAct Tool Schemas)

Agent 内部使用的工具集, 以 Linux 命令命名, 兼容 OpenAI function-calling 格式:

| 工具名 | 说明 | 底层操作 |
|--------|------|---------|
| `cat` | 读取文件内容 | `read_file` |
| `write_file` | 写入/创建文件 (echo > file) | `write_file` |
| `append_file` | 追加内容 (echo >> file) | `patch_file(append=)` |
| `sed` | 查找替换 (sed s/find/replace/) | `patch_file(find=, replace=)` |
| `rm` | 删除文件 | `delete_file` |
| `cp` | 复制文件 | `read_file` + `write_file` |
| `mv` | 移动/重命名 | `read_file` + `write_file` + `delete_file` |
| `touch` | 创建空文件 | `write_file("")` |
| `ls` | 列出所有文件 | `list_files` |
| `find` | 按 glob 搜索路径 | `find_files` |
| `grep` | 按正则搜索内容 | `grep_files` / `read_file` + regex |
| `diff` | 比较两个文件 | `read_file` × 2 + `unified_diff` |
| `head` | 显示前 N 行 | `exec_run("head")` |
| `tail` | 显示后 N 行 | `exec_run("tail")` |
| `wc` | 行/词/字符统计 | `exec_run("wc")` |
| `sort_file` | 排序 | `exec_run("sort")` |
| `commit` | 提交暂存变更 (git commit) | `commit_workspace` |
| `rollback` | 丢弃暂存变更 (git checkout) | `rollback_staged` |
| `log` | 提交历史 (git log) | `log_workspace` |
| `snapshot` | 创建命名快照 (git tag) | `create_snapshot` |
| `script` | 执行 T2Script 批量操作 | `exec_script` |
| `python_exec` | 执行 Python 代码 (CWD=工作区) | FUSE/materialize + subprocess |
| `shell_exec` | 执行 Shell 命令 (CWD=工作区) | FUSE/materialize + subprocess |

### 6.5 FUSE 代码执行架构

Agent 不仅能调用 Linux 风格命令, 还能编写并运行 Python/Shell 代码. 代码的文件 I/O 透明地映射到工作区数据库:

```
用户: "用 Python 统计 data.csv 的词频"
  → Agent 调用 python_exec(code="with open('data.csv') as f: ...")
    → subprocess(python3, cwd=/tmp/t2-fuse/main/)
      → FUSE 拦截 open("data.csv")
        → WorkspaceDB.read_file("main", "data.csv")
          → SQLite 查询 → 返回内容
```

**核心组件**:

| 组件 | 文件 | 职责 |
|------|------|------|
| WorkspaceFUSE | `workspace_fuse.py` | FUSE Operations 实现, 将 POSIX 系统调用映射到 WorkspaceDB |
| FuseManager | `workspace_fuse.py` | 管理 per-workspace 挂载, 按需创建 daemon thread |
| MaterializeFallback | `workspace_fuse.py` | FUSE 不可用时: 导出 → 执行 → diff 回写 |
| CodeExecutor | `code_executor.py` | subprocess 执行器, CWD=挂载点, timeout + 输出截断 |

**FUSE 操作映射**:

| POSIX 调用 | FUSE 方法 | WorkspaceDB 调用 |
|-----------|-----------|-----------------|
| `open(path, "r")` | `read()` | `read_file(ws, path)` |
| `open(path, "w")` | `write()` + `flush()` | `write_file(ws, path, content)` (暂存区) |
| `os.listdir(dir)` | `readdir()` | `list_files(ws)` 按前缀过滤 |
| `os.stat(path)` | `getattr()` | `list_files(ws)` (带元数据缓存, TTL=2s) |
| `os.remove(path)` | `unlink()` | `delete_file(ws, path)` |
| `os.rename(a, b)` | `rename()` | `read + write + delete` |

**安全边界**:
- subprocess 强制 timeout (默认 30s, 最大 120s)
- 输出截断 (50,000 字符)
- Docker 容器级 memory/cpu 限制
- FUSE 仅暴露单个 workspace (无法跨 workspace 访问)

**降级策略**: 当 FUSE 不可用 (无 libfuse / macOS 未装 macFUSE) 时, 自动切换为 "materialize" 模式:
1. 将 workspace 文件导出到临时目录
2. subprocess 以临时目录为 CWD 执行代码
3. 执行后 diff 临时目录, 变更写回 DB
4. 清理临时目录

对 Agent 完全透明 -- `python_exec` / `shell_exec` 接口不变.

### 6.6 Agent 模式与降级策略

系统支持两种模式, 运行时根据 LLM 配置自动选择:

| 能力 | ReAct Agent (有 LLM) | Command Agent (无 LLM, fallback) |
|------|----------------------|----------------------------------|
| 自然语言任务 ("把 a 和 b 合并到 c") | ✅ 自主规划+执行 | ❌ 提示需配置 LLM |
| 多步骤复杂任务 | ✅ 最多 12 轮推理 | ❌ |
| 直接输入 Linux 命令 (cat/grep/cp) | ✅ | ✅ 正则解析执行 |
| 代码编写+执行 (Python/Shell) | ✅ (FUSE/materialize) | ❌ |
| T2Script 脚本 | ✅ | ✅ |
| Web 搜索 | ✅ (需 BRAVE_API_KEY) | ❌ |

**配置方式**:
- 设置 `LLM_API_KEY` (或 `GLM_API_KEY` / `ZHIPUAI_API_KEY`) → 启用 LangGraph ReAct Agent
- 可选: `LLM_PROVIDER=openai` (默认, OpenAI-compatible) 或 `zhipuai` (ChatZhipuAI)
- 可选: `LLM_BASE_URL` 切换端点 (DeepSeek, Moonshot, 本地模型等)
- 可选: `LLM_MODEL` 指定模型名 (默认 glm-4-flash)
- 可选: `LANGFUSE_PUBLIC_KEY` + `LANGFUSE_SECRET_KEY` + `LANGFUSE_HOST` → 启用 Langfuse 可观测性
- 可选: `BRAVE_API_KEY` → 启用 Web 搜索能力
- 未配置 LLM 时自动降级为 Command Agent (仅支持直接命令输入)

**设计原则**: LangGraph ReAct Agent 是主要交互路径, 用户全程自然语言; Command Agent 仅作为零外部依赖的降级方案.

### 6.6 可观测性 (Langfuse)

当配置了 `LANGFUSE_PUBLIC_KEY` / `LANGFUSE_SECRET_KEY` / `LANGFUSE_HOST` 后, 系统自动将每次 Chat 请求追踪到 Langfuse:

- **LLM 调用**: 每轮 model 请求的 input/output、token 用量、延迟
- **工具执行**: 每个 @tool 的输入参数和返回值
- **端到端 Trace**: 从用户消息到最终回复的完整链路
- **成本统计**: 按模型自动计算 token 成本

集成方式: 通过 `langfuse.langchain.CallbackHandler` 注入到 LangGraph 的 `config["callbacks"]`, 对业务代码零侵入.

### 6.6 搜索操作详解

#### `fs.find` -- 路径搜索

基于 `fnmatch` glob 模式搜索文件路径. 在工作区快照(已提交 + 暂存叠加)上执行.

```
支持的 glob 语法:
  - *.md       → 匹配所有 .md 文件
  - docs/*.py  → 匹配 docs 下的 .py 文件
  - *test*     → 匹配路径中包含 test 的文件

触发方式:
  - CLI: t2 fs.find "*.md"
  - API: GET /api/find?workspace=main&pattern=*.md
  - Chat: /find *.md  |  查找 *.py  |  搜索文件 *.txt
  - LLM Tool: fs.find(pattern="*.md")
  - T2Script: fs.find "*.md"
```

#### `fs.grep` -- 内容搜索

基于 Python `re` 正则表达式搜索文件内容. 返回匹配行及行号.

```
支持的正则语法: Python re 模块完整语法
  - import os         → 精确文本匹配
  - TODO|FIXME        → 多词匹配
  - def \w+\(         → 函数定义匹配
  - (?i)error         → 忽略大小写

触发方式:
  - CLI: t2 fs.grep "import os"
  - API: GET /api/grep?workspace=main&pattern=TODO
  - Chat: /grep TODO  |  搜索 import  |  grep "def \w+"
  - LLM Tool: fs.grep(pattern="TODO", max_matches=200)
  - T2Script: fs.grep "TODO"  或管道内  fs.read "file.py" | grep "import"

限制:
  - max_matches 默认 200, 防止大量匹配导致响应过大
  - 仅搜索文本文件(blob 以 UTF-8 解码)
  - 搜索范围是工作区快照, 不是磁盘文件
```

## 8. 并发模型

### 8.1 SQLite WAL 模式

POC 阶段使用 SQLite WAL (Write-Ahead Logging):
- 支持并发读取(多个 Agent 同时读取不同工作区)
- 单写者串行化(写操作通过 SQLite 内部锁排队)
- 每个高级操作是一个原子事务(连接上下文管理器)

### 8.2 工作区级隔离

- 每个 Agent 在自己的工作区中操作
- 暂存区按工作区隔离
- 读取走快照(已提交数据 + 本工作区暂存), 不受其他工作区影响
- 合并是唯一的跨工作区操作, 通过乐观冲突检测保证一致性

### 8.3 冲突检测策略

- 路径级粒度: 同一路径被两个工作区修改 → 冲突
- 乐观策略: 只在合并时检测, 不在写入时加锁
- 相同操作相同内容: 不视为冲突(幂等)

### 8.4 扩展到分布式

替换 SQLite 为 FoundationDB/TiKV 后:
- 元数据层: 分布式事务 KV, 支持多写者
- 内容层: 对象存储(S3/MinIO), 无限水平扩展
- 事件层: Kafka/NATS, 支持流式消费
- 热点缓解: 路径级/段级冲突检测 + append-friendly object store

## 11. 存储引擎规格

### 11.1 连接管理

```python
@contextmanager
def _connect(self) -> sqlite3.Connection:
    conn = sqlite3.connect(self.db_path)
    conn.row_factory = sqlite3.Row
    conn.execute("PRAGMA foreign_keys = ON")
    conn.execute("PRAGMA journal_mode = WAL")
    try:
        yield conn
        conn.commit()
    except Exception:
        conn.rollback()
        raise
    finally:
        conn.close()
```

每个高级操作获取独立连接, 操作完成后自动提交或回滚. 这保证了:
- 无长寿命连接(避免锁持有时间过长)
- 异常安全(所有失败路径回滚)
- WAL 模式下的并发读取

### 11.2 快照重建

读取路径通过 commit chain 回溯重建:

```
HEAD commit → parent → parent → ... → root
         ↓           ↓           ↓
    commit_changes  commit_changes  commit_changes
```

对每个路径, 取最近一次出现的操作(upsert 或 delete). 然后叠加暂存区变更.

### 11.3 内容寻址

```
content → SHA-256(content) → blob_hash
blob_hash → blobs 表 → 实际内容
```

同一 blob_hash 只存储一次, 无论被多少文件/提交引用.

### 11.4 缓存层

```
请求 → WorkspaceCache → CacheBackend(Redis/LRU) → hit? → 返回
                                                  → miss? → SQLite → 写回缓存 → 返回
```

三级缓存架构:

| 层级 | 缓存键 | 失效策略 | TTL |
|------|--------|---------|-----|
| L1 blob cache | `t2:blob:{hash}` | 永不失效(内容寻址) | 24h |
| L2 file cache | `t2:file:{ws}:{path}` | write/delete/commit/rollback 时失效 | 默认 |
| L3 state cache | `t2:state:{ws}:{kind}` | 工作区级任何变更时失效 | 默认 |

后端自动选择:
- **Redis** (`T2_REDIS_URL` 环境变量): 适合多进程/多节点部署, SCAN 前缀清除
- **本地 LRU** (默认): 零依赖, 单进程内 `OrderedDict + threading.Lock`, 上限 4096 条目

缓存失效时机:
- `write_file` → 失效 L2(该路径) + L3(该工作区)
- `delete_file` → 同上
- `commit_workspace` → 失效整个工作区(L2 + L3)
- `rollback_staged` → 同上
- `merge_workspace` → 失效源和目标两个工作区
- `reset_workspace` → 失效该工作区

核心设计原则:
1. **L1 不失效**: blob 是 SHA-256 内容寻址, 同一 hash 永远对应同一内容
2. **写时失效, 非写时更新**: 避免 cache stampede, 下次读自然回填
3. **容错降级**: 缓存后端异常 → 静默降级为 cache miss, 不阻断业务

API 端点:
- `GET /api/cache-stats` → 返回缓存后端类型、命中率、当前条目数

## 15. 关键设计决策记录 (ADR)

### ADR-001: CLI 而非 SQL

**决策**: 用 `fs.read`, `ws.commit` 等命令动词而非 SQL 语句作为 Agent 接口.

**原因**: LLM 已经具备 shell/CLI 调用能力, 迁移成本最低. SQL 会引入语法解析复杂性和安全风险(注入). CLI 动词集可以精确控制 Agent 的操作边界.

### ADR-002: 内容寻址而非路径寻址

**决策**: 文件内容以 SHA-256 哈希为键存储, 而非以路径为键.

**原因**: 自动去重(多个 Agent 写入相同内容只存一份); 不可变(已存储的 blob 永不修改); 便于验证(客户端可校验哈希).

### ADR-003: 暂存区 + 原子提交

**决策**: 所有写操作先进暂存区, 显式提交后才可见.

**原因**: 保证 Agent 的半成品工作不影响其他读者; 支持撤回(rollback); 提交时可整体评估策略钩子.

### ADR-004: 白名单 exec 而非通用 shell

**决策**: `exec.run` 只允许预定义的文本处理命令, 不暴露通用 shell.

**原因**: 通用 shell 执行需要完整进程隔离(容器/VM), 这正是本项目试图避免的开销. 白名单命令在用户态纯 Python 中执行, 读取工作区快照而非磁盘, 无副作用.

### ADR-005: T2Script 进程内解释器替代 Docker/VM

**决策**: 设计并实现 T2Script -- 一个 shell-like 的进程内脚本语言, 让 Agent 编写和执行脚本, 而非使用 Docker/VM 运行 Python/Shell.

**原因**:
- 工作区文件系统已虚拟化为 SQL, 执行引擎也应虚拟化为进程内解释器
- Docker/VM 每 Agent 需 ~512MB RAM, T2Script 每 Agent 仅需 ~2MB, 并发能力从 ~10 提升到 ~1000+
- 语言级隔离(无 syscall, 无真实 FS 访问)天然安全, 无需 OS 级沙箱
- 执行边界(步数/时间/调用深度/HTTP 次数)精确可控
- 工作区事务天然支持回滚, 无需 VM 快照

**实现要点**:
- Lexer → Parser → AST → 树遍历解释器, 纯 Python 标准库
- 管道(`|`)是核心组合机制, 对 LLM 友好
- 38 个内置命令覆盖文本处理/FS/WS/HTTP/数据处理
- 每次执行创建独立的 ScriptContext, 线程安全

### ADR-006: SQLite 作为 POC 后端

**决策**: 使用 SQLite WAL 而非直接上 FoundationDB/TiKV.

**原因**: 零依赖部署; 足以验证 MVCC/事务/冲突检测语义; CLI 接口层与存储层解耦, 后端替换不影响上层. POC 阶段应把精力放在语义正确性而非分布式一致性上.

### ADR-008: LangGraph ReAct Agent + Langfuse 可观测性

**决策**: Agent 从手写 ReAct 循环迁移到 LangGraph `create_react_agent`, 同时接入 Langfuse 作为 LLM 可观测性系统.

**原因**:
- LangGraph 是 LangChain 生态的标准 Agent 框架, `create_react_agent` 封装了 ReAct 循环的最佳实践
- 21 个 `@tool` 函数 (`workspace_tools.py`) 与 Agent 图解耦, 可独立测试和复用
- LangChain ChatModel 抽象统一了多 LLM 厂商接入 (ChatOpenAI / ChatZhipuAI), 不再需要手写 HTTP 客户端
- Langfuse CallbackHandler 零侵入追踪所有 LLM 调用和工具执行, 无需在业务代码中埋点
- 旧的手写循环 (`_LegacyReActAgent`) 保留为 fallback, 当 langgraph 未安装时自动降级

**实现要点**:
- `workspace_tools.py`: 21 个 LangChain `@tool` 函数, 通过闭包绑定 db + workspace
- `graph_agent.py`: `LangGraphAgent` 封装 `create_react_agent` + Langfuse callback + LLM provider 工厂
- `agent.py`: `LLMWorkspaceAgent` 作为适配器, 委托到 `LangGraphAgent` 或 `_LegacyReActAgent`
- 环境变量: `LLM_PROVIDER` / `LLM_API_KEY` / `LLM_BASE_URL` / `LLM_MODEL` + `LANGFUSE_*`

### ADR-007: 三级缓存层 (Redis/LRU 双后端)

**决策**: 在 SQLite 前加三级缓存 -- L1 blob、L2 file、L3 state, 支持 Redis 和本地 LRU 两种后端, 运行时自动选择.

**原因**:
- 直接打 SQLite 在高并发 Agent 场景下有锁竞争风险, 读密集路径(read_file, list_files, tree_workspace)需要缓冲
- L1 利用内容寻址不变性, blob 缓存永不过期(仅 LRU 淘汰), 命中率极高
- L2/L3 采用"写时失效"策略, 避免 cache stampede 和脏读
- Redis 后端为多进程/分布式部署预留扩展, 但 POC 阶段本地 LRU 零依赖即可运行
- 缓存后端异常时静默降级, 不阻断业务流

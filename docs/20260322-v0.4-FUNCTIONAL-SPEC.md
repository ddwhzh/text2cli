# text2cli 功能规格文档

> 版本: 0.4 | 日期: 2026-03-22 | 状态: 已实现

---

## 1. 核心概念

### 1.1 工作区 (Workspace)

工作区是 Agent 的隔离执行域, 类比 Git 的分支. 每个工作区具有:

| 属性 | 说明 |
|------|------|
| `name` | 唯一标识符 (如 `main`, `agent-a`, `task-42`) |
| `head_commit_id` | 当前最新提交的引用 |
| `base_commit_id` | 分叉点, 用于合并时的冲突检测 |
| `tracking_workspace` | 上游工作区 (若通过 fork 创建) |

关键语义:
- 工作区创建是即时的 (仅插入一条引用记录, 不复制数据)
- 多个工作区可同时对同一底层数据进行独立操作
- 每个 Agent 应操作自己的工作区, 避免直接修改 `main`

### 1.2 暂存区 (Staged Changes)

所有写操作首先进入暂存区, 不直接影响已提交的快照:
- 写入可撤回 (ws.rollback)
- 提交是原子的 (全部或无)
- 其他读者不受未提交写入影响

### 1.3 提交 (Commit)

提交是一次原子状态变更:

| 字段 | 说明 |
|------|------|
| `commit_id` | 全局唯一标识 |
| `parent_commit_id` | 父提交引用, 形成链式历史 |
| `kind` | `root` / `commit` / `merge` |
| `message` | 提交说明 |
| `commit_changes` | 此次提交修改的路径列表及操作 (upsert/delete) |

### 1.4 内容寻址存储

文件内容以 SHA-256 哈希为键存储, 相同内容只存一份, 跨工作区/跨提交自动去重.

### 1.5 快照 (Snapshot)

命名快照是对特定提交的不可变引用, 用于标记里程碑、时间旅行读取锚点、审计基线.

### 1.6 事件流 (Events)

所有关键操作自动生成事件记录:

| 事件类型 | 触发时机 |
|---------|---------|
| `workspace_initialized` | 数据库初始化 |
| `workspace_created` | 创建新工作区 |
| `pre_commit` / `post_commit` | 提交前/后 |
| `pre_merge` / `post_merge` | 合并前/后 |
| `merge_conflict` | 检测到合并冲突 |
| `snapshot_created` | 创建命名快照 |
| `staged_rollback` | 回滚暂存变更 |
| `workspace_reset` | 重置工作区 HEAD |

## 2. CLI 命令规格

所有命令返回 JSON. 退出码: 0=成功, 1=校验错误, 2=合并冲突, 3=策略拒绝.

### 2.1 工作区管理

| 命令 | 说明 | 输出 |
|------|------|------|
| `t2 init` | 初始化数据库, 创建 `main` 和根提交 | `{status, db_path, main_head_commit_id}` |
| `t2 ws.create <name> [--from <ws>]` | 从现有工作区分支 | `{status, workspace, from_workspace, base_commit_id}` |
| `t2 ws.list` | 列出所有工作区 | `{status, workspaces: [...]}` |
| `t2 ws.diff [--workspace <name>]` | 暂存区与 HEAD 差异 | `{status, workspace, changes: [...]}` |
| `t2 ws.commit -m <msg>` | 原子提交暂存变更 | `{status, workspace, commit_id, paths}` |
| `t2 ws.merge --source <ws> --target <ws> -m <msg>` | 合并工作区 | `{status, source, target, commit_id}` |
| `t2 ws.log [--limit N]` | 提交历史 | `{status, commits: [...]}` |
| `t2 ws.rollback` | 丢弃暂存变更 | `{status, discarded_count}` |
| `t2 ws.reset --to <commit_id>` | 重置 HEAD 到祖先提交 | `{status, previous_head, new_head}` |
| `t2 ws.snapshot <name>` | 创建命名快照 | `{status, snapshot, commit_id}` |
| `t2 ws.snapshots` | 列出快照 | `{status, snapshots: [...]}` |

### 2.2 文件操作

| 命令 | 说明 |
|------|------|
| `t2 fs.read <path> [--at <commit_id>]` | 读取文件 (支持时间旅行) |
| `t2 fs.write <path> --text <content>` | 写入/创建文件到暂存区 |
| `t2 fs.patch <path> --find <text> --replace <text>` | 查找替换 |
| `t2 fs.patch <path> --append <text>` | 追加内容 |
| `t2 fs.delete <path>` | 删除文件 |
| `t2 fs.list` | 列出工作区所有文件 |
| `t2 fs.find <glob>` | 路径搜索 |
| `t2 fs.grep <regex>` | 内容搜索 |

### 2.3 受控执行

| 命令 | 说明 |
|------|------|
| `t2 exec.run <command> [--path <path>]` | 白名单文本处理命令 (wc/sort/head/tail/uniq/tr/toc/wordfreq) |
| `t2 exec.script --code <T2Script>` | 执行 T2Script 脚本 |

## 3. HTTP API 规格

所有端点前缀 `/api/`, 返回 JSON.

### 3.1 系统端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/health` | 健康检查 `{status: "ok"}` |
| GET | `/api/config` | 运行配置 (LLM 模型/搜索状态) |
| GET | `/api/tool-schemas` | 导出 LLM function-calling 工具 schema |
| GET | `/api/cache-stats` | 缓存统计 (后端/命中率/条目数) |

### 3.2 工作区端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/workspaces` | 列出工作区 |
| POST | `/api/workspaces` | 创建工作区 `{name, from_workspace?}` |
| GET | `/api/state?workspace=` | 聚合状态 (文件/diff/提交/事件) |
| POST | `/api/commit` | 提交暂存 `{workspace, message}` |
| POST | `/api/rollback` | 回滚暂存 |
| POST | `/api/snapshot` | 创建快照 `{workspace, name}` |
| GET | `/api/snapshots` | 列出快照 |

### 3.3 文件端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/file?workspace=&path=` | 读取文件 |
| PUT | `/api/file` | 写入文件 `{workspace, path, content}` |
| PATCH | `/api/file` | 修补文件 `{workspace, path, find?, replace?, append?}` |
| DELETE | `/api/file?workspace=&path=` | 删除文件 |
| GET | `/api/find?workspace=&pattern=` | 路径搜索 |
| GET | `/api/grep?workspace=&pattern=` | 内容搜索 |
| GET | `/api/tree?workspace=` | 目录树视图 |

### 3.4 Agent 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/chat` | Agent 对话 (同步, 返回完整结果) |
| POST | `/api/chat-stream` | Agent 对话 (SSE 流式, token 级实时推送) |
| POST | `/api/exec` | 受控命令执行 |
| POST | `/api/exec-script` | T2Script 脚本执行 |

### 3.5 SSE 事件协议 (`/api/chat-stream`)

使用 Server-Sent Events 协议, 每个事件格式: `event: <type>\ndata: <json>\n\n`

| 事件类型 | 说明 | data 结构 |
|---------|------|-----------|
| `token` | LLM 生成的文本 token (实时逐字) | `string` (token 文本片段) |
| `tool_call` | Agent 决定调用工具 | `{tool: string, args: object}` |
| `tool_result` | 工具执行结果 | `{tool: string, status: "ok"/"error", output: string}` |
| `error` | 执行错误 | `string` (错误信息) |
| `done` | 执行完成 | `{status, reply, actions, llm_rounds}` |

典型事件流:

```
event: token        # LLM 开始思考, 逐字输出
data: "我来"
event: token
data: "帮你"
event: token
data: "搜索..."
event: tool_call    # 决定调用工具
data: {"tool":"web_search","args":{"query":"比特币行情"}}
event: tool_result  # 工具返回
data: {"tool":"web_search","status":"ok","output":"1. Bitcoin..."}
event: token        # LLM 继续生成
data: "根据搜索结果..."
event: done         # 全部完成
data: {"status":"ok","reply":"...","actions":[...],"llm_rounds":3}
```

### 3.6 错误码

| HTTP 状态码 | 含义 |
|------------|------|
| 200 | 成功 |
| 400 | 校验错误 / 未找到 |
| 403 | 策略拒绝 |
| 404 | 路由不存在 |
| 409 | 合并冲突 (附冲突列表) |

### 4.2 代码执行工具 (2 个, 需 FUSE)

需要 macFUSE/libfuse 已安装. 未安装时这两个工具不会注册.

| 工具名 | 说明 | 实现 |
|--------|------|------|
| `python_exec` | 执行 Python 代码, CWD=工作区 FUSE 挂载点 | `CodeExecutor.execute("python", code)` |
| `shell_exec` | 执行 Shell 命令, CWD=工作区 FUSE 挂载点 | `CodeExecutor.execute("shell", command)` |

#### 4.2.1 执行模式与最小硬化

代码执行属于高风险能力. text2cli 将其分为 2 种运行模式(仅影响代码执行工具, 不影响 21 个基础工具与 T2Script):

- `local-trusted` (默认): 单机单用户, 允许在宿主机直接执行 subprocess.
  - 仍会做最小硬化: 默认剔除敏感环境变量(LLM/Langfuse/Brave/Redis key 等), 并对输出大小做硬上限截断.
- `local-untrusted`: 本机也按不可信代码治理.
  - 默认禁止 `exec_backend=host` 在宿主机执行(避免产生“看似隔离, 实则可逃逸”的误导).
  - 仅允许切到 `exec_backend=docker`(Linux 优先) 或未来 microVM/远程 sandbox 后端后再启用执行.
  - docker 后端默认 `--network none` 禁网 + `--cap-drop ALL` + `--read-only` + `tmpfs /tmp` + `pids-limit/ulimit` 等最小隔离与资源边界.

### 4.3 搜索工具 (1 个, 需 Brave API key)

需要配置 `BRAVE_API_KEY`. 未配置时此工具不会注册.

| 工具名 | 说明 | 实现 |
|--------|------|------|
| `web_search` | 通过 Brave Search API 搜索互联网 | `BraveSearchClient.search(query, count)` |

### 4.4 工具总数

| 条件 | 工具数 |
|------|--------|
| 基础 (无 FUSE, 无 Brave) | 21 |
| + Brave 搜索 | 22 |

## 5. T2Script 脚本引擎

### 5.1 设计动机

`exec.run` 仅支持单条命令, Agent 需多次 LLM 往返完成批量操作. T2Script 是进程内 shell-like 脚本引擎, 每 Agent 仅需 ~2MB RAM.

### 5.2 语法

```bash
# 变量与类型: string, number, bool, list, map, null
let name = "project"
let items = ["a", "b", "c"]
let config = {host: "localhost", port: 8080}

# 管道 -- 核心组合机制
fs.read "data.csv" | sort | head 5 | fs.write "top.csv"

# 字符串插值
echo "hello $name"

# 控制流
if $count > 10 { echo "big" } else { echo "small" }
for file in fs.find "*.md" { echo "$file" }

# 函数
fn summarize(path) {
    let content = fs.read $path
    return {path: $path, words: $content | wc -w}
}

# 异常处理
try { fs.read "missing.txt" } catch err { echo "Not found: $err" }

# 模块导入
import "lib/helpers.t2"
```

### 5.3 内置命令 (38 个)

| 类别 | 命令 |
|------|------|
| 文本处理 | `echo`, `grep`, `sort`, `uniq`, `head`, `tail`, `wc`, `tr`, `replace`, `split`, `join`, `cut`, `upper`, `lower`, `trim`, `rev`, `toc`, `wordfreq` |
| 数值聚合 | `sum`, `avg`, `min`, `max`, `count` |
| 数据处理 | `json.get`, `json.pick`, `csv.col`, `format` |
| FS 操作 | `fs.read`, `fs.write`, `fs.append`, `fs.delete`, `fs.list`, `fs.tree`, `fs.find`, `fs.grep`, `fs.exists` |
| WS 操作 | `ws.commit`, `ws.diff`, `ws.rollback`, `ws.snapshot`, `ws.log` |
| 网络 | `web.search`, `http.get`, `http.post` |

### 5.4 执行约束

| 约束 | 默认值 |
|------|--------|
| max_steps | 10,000 |
| max_time | 30s |
| max_call_depth | 50 |
| max_http_calls | 10 |
| max_output | 1MB |

## 6. 策略钩子

### 6.1 钩子类型

| 类型 | 说明 | 配置示例 |
|------|------|---------|
| `path_deny` | 拒绝匹配 glob 的路径 | `{"pattern": "*.secret"}` |
| `max_file_size` | 拒绝超大文件 | `{"max_bytes": 1048576}` |
| `require_message_prefix` | 强制提交消息前缀 | `{"prefix": "feat: "}` |

### 6.2 行为

- 评估时机: `pre_commit` 和 `pre_merge`
- 匹配范围: `workspace_pattern` (glob), 默认 `*`
- 拒绝行为: 抛出 `PolicyRejection`, 整个事务回滚, HTTP 403

## 7. 前端 UI 规格

### 7.1 布局

```
+---------------------------------------------------+
|              标题栏 (Title Bar)                      |
|  Logo | 工作区选择 | + File | Fork | Diff | Commit  |
+----------+--------------------+-------------------+
|          |                    |                   |
| 侧边栏    |    编辑器区域        |   Agent Chat      |
| 240px    |    (flex)           |   340px          |
|          |                    |                   |
| EXPLORER |  Tabs + Editor      |  消息流            |
| COMMITS  |  with line numbers  |  token 流式       |
| EVENTS   |  or Diff View       |  工具调用步骤       |
|          |                    |  输入框            |
+----------+--------------------+-------------------+
|              状态栏 (Status Bar)                      |
|  workspace | HEAD | 文件数 | staged | LLM 信息       |
+---------------------------------------------------+
```

### 7.2 Agent Chat 交互流程

1. 用户在输入框输入自然语言, 按 Enter 发送
2. 前端 POST `/api/chat-stream`, 建立 SSE 连接
3. `token` 事件: 紫色边框区域实时逐字显示 LLM 输出
4. `tool_call` 事件: 累积的文本折叠为 "Thinking", 显示工具调用 `⚙ web_search(query=...)`
5. `tool_result` 事件: 显示结果 `✓ 1. Bitcoin...` 或 `✗ Error: ...`
6. 若为修改型工具 (write_file/rm/commit 等), 自动刷新文件树
7. `done` 事件: 最终回复渲染为 reply 气泡, 状态栏显示 "done"

### 7.3 设计语言

- 色系: Catppuccin Mocha 暗色主题
- 字体: JetBrains Mono (代码), Inter (UI)
- 图标: Unicode 字符 (无外部图标库)
- 流式动画: 紫色左边框脉冲 (token-pulse), 150ms ease 过渡

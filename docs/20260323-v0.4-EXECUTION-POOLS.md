## Execution Pools & Remote Sandbox (v0.4)

- 文档版本: `20260323-v0.4`
- 目标: 落地 `docs/20260323-v0.4-HARNESS-EVALUATION.md` 的长期路线(7.3): 将高风险执行从宿主机迁移到“执行池”, 并通过“过滤 + 打分 + 绑定”的调度策略选择合适的后端.

---

### 1. 核心概念

- **执行后端(backend)**: 具体的执行机制, 如 `host`/`docker`/`microvm`/`remote`.
- **执行池(pool)**: 一组同类型后端的运行资源集合, 具有可比较的属性(隔离强度、延迟、成本、网络策略等). 长期可对应多个运行集群/地域/不同权限域.
- **调度(scheduling)**: 给定一次执行请求, 通过 **过滤(filter)** -> **打分(score)** -> **绑定(bind)** 的方式选出一个 pool 并执行.

---

### 2. 运行模式与默认安全策略

- `local-trusted`:
  - 允许 `exec_backend=host` 在宿主机执行.
  - 仍执行最小硬化(敏感 env 剔除、输出硬截断、超时).
- `local-untrusted`:
  - **默认禁止** `exec_backend=host`(避免产生“看似隔离”的误导).
  - 允许 `exec_backend=docker`(Linux 优先)、`exec_backend=microvm` 或 `exec_backend=remote`(远程 sandbox) 执行.

---

### 3. 后端能力矩阵(定性)

| backend | 隔离面 | 默认网络 | 复现性 | 适用场景 |
|--------|--------|----------|--------|----------|
| `host` | 弱 | 跟随宿主机 | 受宿主机漂移影响 | `local-trusted` 本地快速迭代 |
| `docker` | 中(容器) | `none` | 镜像可控 | `local-untrusted`/高风险任务(优先 Linux) |
| `microvm` | 高(microVM) | `none` | 镜像+runtime 可控 | 高风险/不可信代码/无人值守执行池(优先 Linux) |
| `remote` | 中-高(远程隔离域) | `none` | 镜像+远端环境 | 多租户/更强隔离域/资源弹性 |

说明:
- `microvm` 在 v0.4 中以 "Docker alternative runtime" 形态落地(例如 `io.containerd.kata.v2`), 本质是 "VM-backed container runtime"。
- `remote` 的隔离强度取决于远端底座(容器/microVM/VM). 本版本提供 **remote sandbox service** 的接口与参考实现, 默认以 docker 作为远端执行底座; 也支持在远端将 docker runtime 切到 microVM runtime(形成 remote-microVM 执行池)。

---

### 4. 调度策略(过滤 + 打分 + 绑定)

#### 4.1 过滤(filter)

过滤阶段必须剔除不满足硬约束的 pool:

- `local-untrusted` -> 禁止选择 `host`.
- 若请求 `requires_network=false` -> 允许选择 `network=none` 的 pool(默认).
- 若配置了 `T2_EXEC_REMOTE_URL` 才能启用 `remote` pool.
- 若配置了 `T2_EXEC_REQUIRED_RESIDENCY`(如 `local`/`remote`) -> 只保留满足数据驻留约束的 pool.
- 若配置了 `T2_EXEC_REQUIRED_PERMISSION_DOMAIN`(如 `host`/`container`/`microvm`/`remote`) -> 只保留满足权限域约束的 pool.

#### 4.2 打分(score)

对候选 pool 按维度打分(可配置权重):

- 安全优先级(隔离强度更高得分更高)
- 延迟(更低延迟得分更高)
- 成本(更低成本得分更高)
- 命中率/成功率(更高得分更高, 可通过历史统计或运维配置提供)

#### 4.3 绑定(bind)

选择最高分 pool 并绑定到本次执行请求, 输出中记录 `backend/pool` 元信息(用于审计与复盘).

---

### 5. Remote Sandbox Service (参考实现)

#### 5.1 目标

- 以 HTTP API 提供“远程执行面”, 用于 `exec_backend=remote`.
- 服务端负责:
  - 在隔离环境(docker, 未来可替换 microVM)中执行代码
  - 返回 stdout/stderr/exit_code
  - 返回文件变更(changes/deletes), 由调用方写回 WorkspaceDB(唯一真相源)

#### 5.2 API

- `POST /api/execute`
  - Request(JSON):
    - `language`: `"python"` 或 `"shell"`
    - `code`: string
    - `timeout`: int(秒)
    - `files`: `{path: content}` 的 map(UTF-8 文本)
  - Response(JSON):
    - `stdout`, `stderr`, `exit_code`, `timed_out`
    - `changes`: `{path: content}` (新增/修改)
    - `deletes`: `string[]`

#### 5.3 安全默认值(服务端)

- docker 以最小权限运行:
  - `--network none`
  - `--cap-drop ALL`
  - `--security-opt no-new-privileges`
  - `--read-only` + `tmpfs /tmp`
  - `--pids-limit` + `--ulimit` 资源上限

### 6. 配置项(补充)

| 变量 | 说明 |
|------|------|
| `T2_EXEC_BACKEND` | `host`/`docker`/`microvm`/`remote`(或 `auto`) |
| `T2_EXEC_REMOTE_URL` | remote sandbox 服务地址(如 `http://127.0.0.1:9770`) |
| `T2_EXEC_REMOTE_TOKEN` | (可选) remote 调用鉴权 token |
| `T2_EXEC_SCHEDULER` | `0/1` 是否启用“过滤+打分+绑定”自动选择 |
| `T2_EXEC_POOL_WEIGHTS` | (可选) 调度权重, 例如 `security=5,latency=4,cost=2,hit_rate=1` |
| `T2_EXEC_POOL_HIT_RATES` | (可选) 命中率/成功率 hint, 例如 `host=99,docker=98,microvm=97,remote=96` |
| `T2_EXEC_REQUIRED_RESIDENCY` | (可选) 数据驻留约束, 例如 `local`/`remote` |
| `T2_EXEC_REQUIRED_PERMISSION_DOMAIN` | (可选) 权限域约束, 例如 `host`/`container`/`microvm`/`remote` |
| `T2_EXEC_MICROVM_RUNTIME` | (可选) microVM runtime, 例如 `io.containerd.kata.v2`(需在运行 Docker daemon 的机器上安装 shim) |
| `T2_SANDBOX_DOCKER_RUNTIME` | (服务端) remote sandbox 选择的 docker runtimeType, 例如 `io.containerd.kata.v2`(用于 remote-microVM 池) |

### 7. 边界与后续

- 本版本完成:
  - `local-untrusted` 禁止 host 执行(硬约束)
  - docker/microvm/remote 作为可选执行池后端
  - 最小调度策略(过滤+打分+绑定)的工程骨架
- 后续演进:
  - Firecracker-native microVM 执行池(不依赖 Docker runtime), 与现有 microvm backend 并存
  - workspace 内容寻址增量同步(避免全量 files map)
  - 统一执行审计表(执行请求/选择的 pool/资源消耗/结果/变更集)

# text2cli 需求文档

> 版本: 0.4 | 日期: 2026-03-22 | 状态: POC 已验证, 迭代中

---

## 1. 项目背景

### 1.1 问题陈述

当前主流 AI Agent 工作区方案 (Manus, Codex, Devin) 依赖完整 VM/容器沙箱为每个 Agent 提供隔离环境. 这种方案存在根本性的资源效率问题:

- **启动成本高**: VM/容器引导 + 依赖安装, 单任务开销 125ms-5s
- **存储浪费**: 每任务完整文件系统镜像, 无法跨 Agent 共享内容
- **并发瓶颈**: 单机 ~10 Agent 上限, 受限于 VM 内存和 CPU
- **审计困难**: 操作日志依赖外部系统, 无法内建回滚

### 1.2 核心论题

> 数据库伪装成 Agent 所需的最小工作区语义, 而非通用 POSIX 替身.

text2cli 用事务对象图替代真实磁盘目录, 暴露收窄的 CLI 动词集而非完整 shell, 让 AI Agent "以为" 自己在操作文件系统, 实际所有操作运行在 MVCC 快照隔离的事务存储上.

### 1.3 与现有方案对比

| 维度 | 完整 VM 沙箱 (Manus/Codex) | 事务工作区 (text2cli) |
|------|---------------------------|----------------------|
| 隔离方式 | OS 级进程隔离 | 事务级工作区隔离 |
| 启动成本 | VM/容器引导 + 依赖安装 | 零 (创建工作区行记录) |
| 代码执行 | 通用 Shell/Python (需隔离) | FUSE 挂载 + subprocess / T2Script |
| 存储成本 | 每任务完整文件系统镜像 | 内容寻址去重, 跨 Agent 共享 |
| 并发能力 | ~10 Agent/机器 | ~1000+ Agent/机器 |
| 审计能力 | 外部日志 | 内建提交日志 + 事件流 |
| 回滚能力 | 快照/恢复 VM | ws.rollback / ws.reset |

## 2. 目标用户

| 用户角色 | 使用场景 |
|---------|---------|
| AI Agent | 通过 LLM 自然语言操作工作区文件, 执行代码, 搜索信息 |
| Agent 平台运维 | 管理多 Agent 并发访问, 监控操作审计, 设置策略钩子 |
| 开发者 | 通过 CLI/API 集成 text2cli 到自有 Agent 框架 |

---

## 3. 适用场景

- **文档处理**: Markdown/文本创建、编辑、搜索、组织
- **代码生成**: Agent 生成代码后通过工作区提交审核
- **多 Agent 协作**: 多个 Agent 在独立分支上并行工作, 最终合并
- **批量数据处理**: 结构化文件的批量变换和验证
- **审计场景**: 所有操作可追溯、可回放、可回滚
- **联网任务**: Agent 搜索互联网信息, 整理后写入工作区

## 6. 约束

| 约束 | 说明 |
|------|------|
| POC 存储 | SQLite WAL 模式, 单写者串行化 |
| FUSE 依赖 | 代码执行需要 macFUSE (macOS) 或 libfuse (Linux) |
| LLM 依赖 | Agent 自然语言能力需要配置 LLM API key |
| Python 版本 | >= 3.11 |
| 无认证/多租户 | POC 阶段不含用户认证和租户隔离 |
